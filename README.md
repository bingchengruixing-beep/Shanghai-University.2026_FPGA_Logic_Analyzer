
三、 课题实施详述
3.1 上位机各模块设计具体实施方案
上位机系统抛弃了传统的厚重客户端方案，创新性地采用 B/S（Browser/Server）前后端分离架构。以 Python (FastAPI) 作为底层硬件驱动与数据中转服务，以 Vue3 + HTML5 Canvas 作为前端渲染引擎，实现了跨平台、高可用、低延迟的仪器级操作体验。
3.1.1 总体架构设计与原理
PC 端软件整体划分为四个核心模块：硬件通信层、核心调度层、协议解算层与视图渲染层。
 
3.1.2 各核心模块原理与制作过程
	硬件通信与流控模块（数据黑洞与软件阀门）
	原理：由于缺乏 RTS/CTS 硬件流控，下位机在连续模式下会产生不受控的数据洪流。为防止操作系统底层串口 FIFO 缓冲区溢出导致死锁，后端引入了“软件阀门”与“数据黑洞”机制。
	实施：在 Python 端建立永不停歇的守护线程。当阀门关闭时，线程将缓冲区内的数据全部读出并强制丢弃（数据黑洞），确保物理通道畅通；当用户下发采集指令时，系统先行执行 reset_input_buffer() 清刷陈旧数据，随后打开阀门进入有效帧寻呼状态。
	高速矩阵解包模块
	原理：2048 字节的一维数据需转换为 8 通道、深度为 2048 的二维位矩阵，若使用传统的双重 for 循环按位拆解，会导致极高的 CPU 开销。
	实施：采用 C 语言底层的计算库 NumPy。利用 np.unpackbits() 函数直接对内存视图进行快速展开，并配合 .reshape(-1, 1) 与矩阵转置运算，将解包耗时压缩至微秒级。
	高频渲染与视图模块
	原理：DOM 渲染无法支撑几十万个数据点的实时重绘。
	实施：引入原生 HTML5 Canvas 高性能图表库 uPlot。通过编写底层 draw 钩子，实现了波形的阶跃插值渲染，并在 Y 轴扩展了相互独立的 DEC（并行）、UART、SPI 解码显示轨道，采用不同颜色的透明气泡矩形防叠加上色。
	边缘协议解码模块
	原理：为了实现“调参即见”的零延迟体验，避免频繁的网络往返，UART 与 SPI 的时序解算不依赖后端，全部下沉至浏览器端（边缘计算）。
	实施：利用 JS V8 引擎的高效算力，在前台构建基于时间的有限状态机，提取下降沿（Start Bit）与时钟有效沿（Sample Edge），实时抽取总线数据。

3.2 软件编程实现
本节主要论述上位机软件的通信协议定义、核心算法推导以及主程序调度流程。
3.2.1 详尽的通信协议格式
系统采用基于特征码与算术累加和的双向非对称通信协议，确保恶劣电磁环境下的帧边界安全。
1. 下行控制指令帧（PC \rightarrow FPGA）
固定长度为 8 字节，无帧尾。
	Byte 0-1 (帧头): 0xAA 0x55（固定同步字）
	Byte 2 (CMD): 0x01 启动采样，0x02 中止等待，0x03 强制读取。
	Byte 3 (DIV): 采样率分频系数（0x00=50MHz ... 0x05=10kHz）。
	Byte 4 (MODE): 触发模式（0x00=立即采样，0x01=上升沿... 0x05=组合触发）。
	Byte 5 (P1): 触发参数 1（通道号或 Pattern Value）。
	Byte 6 (P2): 触发参数 2（Pattern Mask）。
	Byte 7 (CHK): 校验和。计算公式为 Byte 0 至 Byte 6 的算术累加和取低 8 位，即 CHK=\sum_{i=0}^{6}{Byte_i}\backslash  pmod256。
2. 上行波形数据帧（FPGA \rightarrow PC）
固定长度为 2051 字节。
	Byte 0-1 (帧头): 0x5A 0xA5（异于下行帧头，便于通道隔离识别）。
	Byte 2-2049 (Payload): 2048 字节的 8 通道并行物理电平采样结果。
	Byte 2050 (CHK): 校验和。计算公式为 Byte 2 至 Byte 2049 的算术累加和取低 8 位。后端读取线程采用“滑窗寻呼法”，仅当帧头与校验和双重通过时，才判定有效并实施解包。
3.2.2 核心算法公式推导
1. 测量光标的量纲自适应算法
在界面框选波形时，测得样本点数差 \Delta X。根据协议中的 DIV 映射出的单一采样周期时间 T_s（单位：\mu s），实际时间差 \Delta T 与信号频率 F 的推导公式为：
\Delta T=\Delta X\times T_s\emsp\left(\mu s\right)
F=\frac{{10}^6}{\Delta T}\emsp\left(Hz\right)
代码内嵌入了自适应进位算法，当 \Delta T\ \geq1000\ 时自动进位为 ms 或 s，保证科学读数规范。
2. UART 串行解码状态机算法
前端根据设定的波特率（BaudRate）与当前物理采样率，计算出每个比特理论应占据的离散样本点数：
SamplesPerBit=\frac{{10}^6/T_s}{BaudRate}
算法必须满足 SamplesPerBit\ \geq2 的香农采样防呆约束。状态机寻找到起始位下降沿索引 i 后，数据位采样点位的计算公式为：
SampleIndex_n=i+SamplesPerBit\times1.5+n\times SamplesPerBit\emsp\left(n\in\left[0,7\right]\right)
该公式确保了采样指针能够精确跃过边沿抖动区，直击数据位电平的最中央。

本系统的测试环节旨在验证 PC 端上位机软件与 FPGA 下位机在不同运行模式下的通信稳定性、协议解算准确性以及 UI 交互的鲁棒性。
1. 测试条件与使用设备：
	硬件设备：Windows个人计算机（作为上位机运算节点），FPGA 开发板（搭载 CH340  USB-TTL 串口桥接芯片，已固化包含标准 SPI/UART 测试时序的激励源）。
	测试工具软件：Python 3.10+（运行 FastAPI 后端服务），Google Chrome / Edge（运行 Vue3 + Canvas 视图），UartAssist 5.15（串口调试助手，用于底层 Hex 报文交叉验证）。
	测试规则与方法：采用黑盒功能测试与白盒数据流监控相结合的方式。前端通过切换单次/连续采样模式、改变解码参数，验证视图响应；后端开启控制台日志，严密监控 5A A5 特征码的滑窗对齐情况及操作系统的串口 FIFO 缓冲区积压状态。

系统测试结果与可行性报告的要求是否一致及原因分析： 测试结果与初期可行性报告中的预期目标高度一致。上位机不仅以极低的系统开销（B/S 轻量级架构）替代了传统庞大的 C++ GUI 客户端，还超额验证了在低速、无握手流控的恶劣硬件环境中，利用软件算法（数据黑洞与主动截断）强行接管并规范硬件行为的可行性。
